# Claude Deck v0.2 Design

Date: 2026-05-20
Status: Approved for planning

## Summary

`v0.2` focuses on making Claude Deck a better inspection shell for `claude` CLI sessions.

This version does not add new agent capabilities, new automation systems, or a separate data store. It improves how existing CLI sessions are replayed, inspected, and searched inside the desktop shell.

The scope is intentionally limited to three feature packages:

1. Structured history replay
2. Full inspector-style right panel
3. Better search-to-context navigation

## Product Goal

Claude Deck is a GUI shell around local `claude` CLI usage. The CLI and on-disk session history remain the source of truth. The app's job is to make that history easier to inspect, resume, search, and understand.

`v0.2` should make historical and live conversations feel much closer in information quality. A user should no longer get rich tool context only while a reply is streaming and lose most of that structure after reopening the same session later.

## Non-Goals

This version will not:

- add a second backend or multi-provider runtime
- introduce a custom session database as the primary source of truth
- add automation/task orchestration features
- redesign the full UI to copy every `codex.app` behavior
- add archive browsing, tab persistence, or project pinning
- build a full file diff workspace beyond current focused previews

## Current Gaps

The current app already supports:

- live structured streaming from `claude` CLI
- tool cards during active streaming
- session replay from local `.jsonl`
- session search across local history

The main problems are:

1. Replay loses structure
   Historical replay largely collapses back into plain text messages on the frontend, even when the original session contained tool activity.

2. Inspector is still shallow
   The right panel shows a lightweight summary but is not yet a true inspector for turns, tools, files, and failures.

3. Search lands on text, not context
   Search can jump to a message, but the surrounding execution context is still weak, especially for tool-heavy turns.

## Scope

### 1. Structured History Replay

#### Objective

Upgrade replay so historical sessions can restore a message structure much closer to live streaming sessions.

#### Expected User Outcome

When reopening a prior conversation, the user can inspect:

- user messages
- assistant text output
- tool usage blocks
- tool results
- todo-style tool progress where available
- replayed usage totals

The reopened session should feel like a preserved work log, not a plain transcript.

#### Design Direction

The data model should move toward a unified message model where live streaming and replayed history can both render into the same conversation UI with minimal branching.

The replay pipeline should prefer structural fidelity over perfect recovery of every raw CLI line. The first version only needs to faithfully restore the common high-value artifacts that matter for inspection.

#### Replay Boundaries

`v0.2` replay should cover:

- text messages
- tool invocation identity and tool name
- tool input when recoverable from stored content
- tool result output when recoverable
- message ids for search targeting
- accumulated usage stats already derived from history

`v0.2` replay does not need to fully recreate:

- every intermediate partial delta
- every internal streaming boundary from the live event flow
- a pixel-perfect clone of the original live render timing

### 2. Inspector-Style Right Panel

#### Objective

Turn the right panel into a real inspection surface rather than a lightweight summary widget.

#### Expected User Outcome

For the active conversation, the user can quickly inspect:

- current turn or latest turn status
- tool steps in order
- tool input and output
- related files touched or referenced
- stderr or failure hints where present
- token and timing context

#### Panel Structure

The first version should use a simple three-section layout:

1. Turn Summary
   Status, elapsed time where available, token summary, error summary

2. Tool Steps
   Ordered list of tool executions with expandable detail

3. Related Files
   Deduplicated file list for the relevant conversation context

This should be built on top of the unified message model rather than by ad hoc rescanning of raw text wherever possible.

#### Interaction Model

The panel should primarily reflect the current conversation state and most relevant visible context. `v0.2` does not need a complex multi-filter inspector with global indexing, cross-session aggregation, or advanced sorting controls.

### 3. Search-To-Context Navigation

#### Objective

Make search results land the user in a richer context, not just on an isolated text node.

#### Expected User Outcome

When a search result is opened, the app should:

- navigate to the correct session
- jump to the correct message or best matching anchor
- preserve enough structure nearby that the user can understand the related execution context

#### Design Direction

Search should continue to read from local `.jsonl` files. This version improves how results are surfaced and consumed, not where they are stored.

The first version should keep the existing search query model but improve the experience after selection:

- stable jump targeting by stored message id or equivalent anchor
- stronger linkage between the selected hit and the inspector panel
- better visibility of nearby tool context after landing

`v0.2` does not add advanced search syntax, structured query filters, or tool-name-specific query grammar.

## Cross-Cutting Design Decisions

### Source of Truth

Local `claude` CLI output and its session files remain the source of truth.

Claude Deck may derive UI-ready structures in Rust or TypeScript, but those are view models only. The app should not create a competing canonical store for session history in this version.

### Model Unification

The preferred direction is:

- live session rendering and replayed session rendering should converge on the same frontend message shape
- the conversation UI and inspector should consume shared message semantics

This reduces UI branching and keeps replay quality aligned with live rendering improvements.

### Incremental Fidelity

If a replayed artifact cannot be reconstructed perfectly, the app should still preserve the highest-value structure rather than falling back to a fully flattened transcript.

The priority order is:

1. Message identity and navigation
2. Tool visibility
3. Tool result visibility
4. Nicely formatted secondary details

## Implementation Boundaries

The implementation can change internal Rust replay parsing and frontend replay loading behavior, but should avoid widening product scope beyond the three approved packages.

Likely change areas include:

- Rust replay parsing and replay response shape
- frontend replay loading and message normalization
- right panel rendering and data derivation
- search landing and related context handling

This design intentionally does not prescribe the exact internal type names or file-level refactor shape. That belongs in the implementation plan.

## Risks

1. Replay fidelity may vary across older session formats
   Mitigation: target common, recent CLI output patterns first and degrade gracefully.

2. UI model drift between live and replay paths
   Mitigation: unify around shared message semantics rather than separate render logic.

3. Search landing may still feel partial if context grouping is weak
   Mitigation: ensure the inspector can show nearby structured context after jump.

## Acceptance Criteria

`v0.2` is complete when all of the following are true:

1. Opening a history session that contains tool activity shows structured tool-related information in the conversation and inspector views.
2. A session viewed live and then reopened later preserves broadly similar information hierarchy for text, tools, and results.
3. Search can jump into a matching session and land on a stable message anchor with surrounding context that is meaningfully inspectable.
4. `pnpm build` passes.
5. `cargo check` passes.

## Suggested Delivery Order

1. Structured replay model
2. Frontend replay normalization
3. Inspector panel upgrade
4. Search landing and context polish

This order minimizes rework because the panel and search behavior both depend on better replay structure.
