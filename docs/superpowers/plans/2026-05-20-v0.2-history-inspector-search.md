# Claude Deck v0.2 History Inspector Search Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Upgrade Claude Deck's history replay, right-side inspector, and search landing so reopened `claude` CLI sessions are inspectable with nearly the same information hierarchy as live sessions.

**Architecture:** Keep local `.jsonl` files as the only source of truth, expand the Rust replay/search payloads to preserve higher-value structure, then normalize that payload into the existing frontend message model plus a small inspector derivation layer. Do not add a new data store or a frontend test runner in this version; use targeted Rust parsing tests, `pnpm build`, `cargo check`, and manual `pnpm tauri dev` verification.

**Tech Stack:** Tauri 2, Rust, React 19, TypeScript, Vite, local `claude` CLI session history

- [ ] **Step 3: Create a replay normalization helper**

Create `src/lib/replay.ts` with pure helpers that convert replay entries into the existing frontend conversation model:

```ts
export type NormalizedReplay = {
  messages: ChatMessage[];
  stderr: string[];
};

export function normalizeReplay(result: ReplayResult): NormalizedReplay {
  const messages: ChatMessage[] = [];
  const toolIndex = new Map<string, number>();

  for (const entry of result.entries) {
    if (entry.is_meta) continue;
    if (entry.kind === "text") {
      messages.push({
        kind: "text",
        id: entry.id,
        role: entry.role === "assistant" ? "assistant" : "user",
        text: entry.text,
      });
      continue;
    }
    if (entry.kind === "tool_call") {
      toolIndex.set(entry.tool_use_id ?? entry.id, messages.length);
      messages.push({
        kind: "tool",
        id: entry.id,
        toolUseId: entry.tool_use_id ?? entry.id,
        role: "tool",
        tool: {
          toolUseId: entry.tool_use_id ?? entry.id,
          name: entry.tool_name ?? "Unknown",
          partialJson: entry.tool_input_json ?? "",
          input: safeParse(entry.tool_input_json),
          output: null,
          isError: false,
          status: "done",
        },
      });
      continue;
    }
    if (entry.kind === "tool_result") {
      // attach to the matching tool card when possible
    }
  }

  return { messages, stderr: [] };
}
```

Implementation notes:

- Prefer reusing `ToolCall` and `ChatMessage` from `src/lib/chatTypes.ts`.
- Parse `tool_input_json` with a safe helper.
- Keep the same filtering rules for meta/local-command noise.
- Preserve stable ids so search highlight can target replayed rows.

```ts
export type InspectorSummary = {
  status: "idle" | "thinking" | "streaming" | "error";
  latestError: string | null;
  stderrCount: number;
  toolCount: number;
  relatedFiles: string[];
  inputTokens: number;
  outputTokens: number;
  contextWindow: number;
};

export function buildInspectorData(args: {
  messages: ChatMessage[];
  status: ConvStatus;
  error: string | null;
  stderr: string[];
  usage: UsageStats;
}) {
  const tools = args.messages.filter((m) => m.kind === "tool");
  const relatedFiles = collectInspectorFiles(tools);
  return {
    summary: {
      status: args.status,
      latestError: args.error,
      stderrCount: args.stderr.length,
      toolCount: tools.length,
      relatedFiles,
      inputTokens: args.usage.inputTokens,
      outputTokens: args.usage.outputTokens,
      contextWindow: args.usage.contextWindow,
    },
    tools,
    relatedFiles,
  };
}
```
